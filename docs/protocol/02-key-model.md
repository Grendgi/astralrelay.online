# Модель ключей и криптография

Версия протокола: **0.1**

## 1. Обзор

- **X3DH** — согласование ключей при первой отправке (forward secrecy, post-compromise)
- **Double Ratchet** — шифрование потока сообщений с обновлением ключей
- Все ключи генерируются на клиенте; сервер хранит только **публичные** данные

---

## 2. Типы ключей

### 2.1 Identity Key

- **Кривая**: Ed25519 (подпись) + X25519 (DH) — или одна комбинированная пара
- **Хранение**: приватный — только на устройстве; публичный — в Key Directory
- **Использование**: идентификация устройства, подпись prekeys

### 2.2 Signed Prekey

- **Кривая**: X25519
- **Подпись**: Ed25519 identity key
- **Срок жизни**: обновляется периодически (например, раз в неделю) или по требованию
- **Количество**: 1 активный на устройство
- **Сервер хранит**: публичный ключ + подпись

### 2.3 One-time Prekey

- **Кривая**: X25519
- **Особенность**: одноразовый, помечается использованным после X3DH
- **Количество**: N штук (рекомендуется 100–500), пополняются клиентом
- **Сервер хранит**: список публичных ключей, помечает consumed

---

## 3. Prekey Bundle

Набор ключей для установки сессии (X3DH):

```json
{
  "identity_key": "base64_ed25519_public",
  "signed_prekey": {
    "key": "base64_x25519_public",
    "signature": "base64_signature",
    "key_id": 1
  },
  "one_time_prekey": {
    "key": "base64_x25519_public",
    "key_id": 42
  }
}
```

- `one_time_prekey` — опционально; если закончились — используется только signed prekey (хуже post-compromise, но работает)
- `key_id` — для идентификации при обновлении/потреблении

---

## 4. X3DH (упрощённо)

1. Инициатор запрашивает prekey bundle получателя
2. Выполняет DH с: identity_key, signed_prekey, (one_time_prekey)
3. Строит shared secret, из него — session key
4. One-time prekey помечается consumed на сервере
5. Первое сообщение содержит initial key agreement material для получателя

Детали: [Signal X3DH spec](https://signal.org/docs/specifications/x3dh/)

---

## 5. Double Ratchet

- **Root chain**: обновление при получении новых DH-материалов
- **Sending/Receiving chain**: симметричные ключи для каждого сообщения
- **Header**: содержит публичный ключ DH (для следующего root step) и номера сообщений

Сервер не участвует: передаёт только шифртекст. Клиент хранит ratchet state локально.

Детали: [Signal Double Ratchet](https://signal.org/docs/specifications/doubleratchet/)

---

## 6. Шифрование файлов

Для каждого файла:

1. Генерируется случайный **file key** (256-bit, AES-256 key)
2. Файл шифруется: **XChaCha20-Poly1305** (или AES-256-GCM) с file_key + random nonce
3. Nonce хранится/передаётся вместе с ciphertext (12 bytes для GCM, 24 для XChaCha)
4. File key + nonce передаются получателю **внутри E2EE-сообщения** (в ciphertext)

Сервер хранит только blob (ciphertext). Content-URI = `blob:sha256:{hash(ciphertext)}`.

## 7. Жизненный цикл ключей

### Регистрация

1. Клиент генерирует: identity key pair, signed prekey, N one-time prekeys
2. Отправляет `m.key.upload` на сервер
3. Сервер сохраняет в Key Directory

### Запрос bundle

1. Клиент A хочет отправить сообщение клиенту B
2. Запрос `m.key.request` (B — локальный или удалённый)
3. Сервер возвращает prekey bundle (или запрашивает через S2S)
4. При использовании one-time prekey — сервер помечает его consumed

### Ротация

| Ключ | Частота | Триггер |
|------|---------|---------|
| Identity | Не ротируется (без смены устройства) | — |
| Signed prekey | Периодически (7 дней) или по истечении срока | Клиент загружает новый |
| One-time prekeys | По мере потребления | Клиент пополняет при < N/2 |

### Удаление устройства

Клиент отправляет `m.key.revoke` — сервер удаляет все ключи устройства. Сессии с другими клиентами становятся недействительными (получатель увидит ошибку расшифровки).

## 8. Аутентификация клиента (C2S)

Варианты для MVP:

**A. Access Token (JWT или opaque)**

- Регистрация/логин возвращает token
- Клиент передаёт в заголовке: `Authorization: Bearer <token>`
- Сервер проверяет подпись/хранит в whitelist

**B. Ed25519 подпись запросов**

- Клиент подписывает тело + timestamp своим identity key
- Заголовок: `X-Signature: base64(signature)`, `X-Timestamp: unix`
- Сервер проверяет подпись по публичному identity key

Для MVP достаточно **A** (проще). B — для усиленной анонимности (токен не привязан к ключу явно в логах).

---

## 9. Server Signing Key (S2S)

Каждый homeserver имеет ключевую пару (Ed25519):

- **Публичный**: публикуется в `/.well-known/server-key` или в federation discovery
- **Приватный**: хранится на сервере, подписывает исходящие S2S-запросы
- Взаимодействующий сервер проверяет подпись

## 10. Рекомендуемые криптобиблиотеки

| Среда | Библиотека |
|-------|------------|
| Браузер (JS) | libsodium.js, tweetnacl, или WASM (Rust libsignal) |
| Node.js | libsodium.js, @noble/* |
| Сервер (Go) | crypto/ed25519, golang.org/x/crypto |
| Сервер (Rust) | ed25519-dalek, x25519-dalek |

Использовать проверенные реализации, не писать крипту с нуля.
